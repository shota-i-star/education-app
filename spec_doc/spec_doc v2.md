## 2. 解決したい課題

- 誰がどの教育項目をどこまで終えたか、一目でわからない
- テストの点数や進捗を手作業（Excel等）で管理しており、集計が煩雑
- 上司とメンバーで見せる情報を分けたい

---

## 3. 主な機能要件

### メンバー向け画面
- チェックリスト形式で教育項目を完了マークできる
- ステップ／フェーズ形式で段階的に進められる
- テストを受けて点数が記録される
- 過去の等級のコンテンツを閲覧できる

### 上司向け画面
- 全メンバーの進捗を一覧・集計して見られる
- テストの平均点・個人別スコアを確認できる
- 進捗率をグラフで把握できる

### 管理者向け画面
- 大カテゴリ・教育項目・テスト内容の追加・編集・削除ができる
- メンバーの等級を変更できる
- メンバーアカウントの管理ができる

### 共通機能
- ログイン機能あり（役割に応じて画面を切り替え）
- 1つのアカウントに複数の役割を付与できる（例：上司兼管理者）
- 役割の種類：メンバー／上司／管理者
- データはサーバー側に保存（リロードしても消えない）

## 6. データベース設計

### check_progress テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | int8 | 自動採番 |
| user_id | text | ユーザーID |
| item_id | text | チェック項目ID |
| checked | bool | チェック状態 |

**ユニーク制約：** `user_id` + `item_id` の組み合わせ（重複不可）

### test_results テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | int8 | 自動採番 |
| user_id | text | ユーザーID |
| category | text | カテゴリ名（例：新人研修） |
| grade | int | 等級（例：1） |
| score | int | 点数 |
| passed | boolean | 合否 |
| taken_at | timestamp | 受験日時（自動記録） |

### item_id の命名ルール
`grade{等級}_{カテゴリ}_{項目}` の形式

例：
- `grade1_training_phone`（1等級・新人研修・電話対応）
- `grade1_azism_team`（1等級・AZism・チームワーク）

## 8. 教育コンテンツ構成

### 等級（6段階）
0等級 → 1等級 → 2等級 → 3等級 → 4等級 → 5等級

### 大カテゴリ（2個）
- 新人研修
- AZism共通言語

### コンテンツ構成のルール
- 累積式：上の等級になると過去の等級の項目も表示される
- 過去の等級は「閲覧のみ」
- 現在の等級のみ「チェック可能」

### テスト仕様
| 項目 | 内容 |
|------|------|
| 出題形式 | 4択・○×・記述を混在可 |
| 採点 | 4択・○×は自動採点。記述式は管理者が採点 |
| 合否判定 | 管理者が合格基準点を設定（現在は80点） |
| 再受験 | 未定 |

## 11. AIへの進め方の指示

1. **フェーズを守る** — 現在のフェーズ外の実装は提案しない
2. **コードには必ず説明をつける** — 日本語で解説する
3. **専門用語は都度説明する** — 具体例を使って説明する
4. **小さく確認しながら進める** — 小さな単位で動作確認してから次へ
5. **ユーザーに選択肢を示す** — 複数の方法がある場合はメリット・デメリットを説明する
6. **わからない点は聞き返す** — 要件が曖昧な場合は実装前に確認する
