## 2. 解決したい課題

- 誰がどの教育項目をどこまで終えたか、一目でわからない
- テストの点数や進捗を手作業（Excel等）で管理しており、集計が煩雑
- 上司とメンバーで見せる情報を分けたい

---

## 3. 主な機能要件

### メンバー向け画面
- チェックリスト形式で教育項目を完了マークできる
- ステップ／フェーズ形式で段階的に進められる
- テストを受けて点数が記録される
- 過去の等級のコンテンツを閲覧できる

### 上司向け画面
- 全メンバーの進捗を一覧・集計して見られる
- テストの平均点・個人別スコアを確認できる
- 進捗率をエリア別・店舗別グラフで把握できる
- メンバー詳細画面で個人の状況を確認できる

### 管理者向け画面
- 大カテゴリ・教育項目・テスト内容の追加・編集・削除ができる
- メンバーの等級を変更できる
- メンバーアカウントの管理ができる
- 記述式テストの採点ができる

### 共通機能
- ログイン機能あり（役割に応じて画面を切り替え）
- 1つのアカウントに複数の役割を付与できる（例：上司兼管理者）
- 役割の種類：メンバー／上司／管理者
- データはサーバー側に保存（リロードしても消えない）
- ログアウト機能あり（全画面共通）

## 6. データベース設計

### user_profiles テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | uuid | Supabase AuthのユーザーIDと紐づく |
| employee_id | text | 社員番号（ログイン時に使用） |
| name | text | 表示名 |
| role | text | 役割（member / boss / admin / boss_admin） |
| grade | int | 等級（0〜5） |
| store | text | 所属店舗 |
| area | text | 所属エリア |

### check_progress テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | int8 | 自動採番 |
| user_id | uuid | ユーザーID |
| item_id | text | チェック項目ID |
| checked | bool | チェック状態 |

**ユニーク制約：** `user_id` + `item_id` の組み合わせ

### test_results テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | int8 | 自動採番 |
| user_id | uuid | ユーザーID |
| category | text | カテゴリ名 |
| grade | int | 等級 |
| score | int | 4択・○×の点数 |
| passed | boolean | 合否 |
| q1_answer | int | 問1で選んだ選択肢番号 |
| q2_answer | text | 問2で選んだ答え（'maru'/'batu'） |
| taken_at | timestamp | 受験日時（自動記録） |

### text_answers テーブル（作成済み）
| カラム名 | 型 | 内容 |
|---------|-----|------|
| id | int8 | 自動採番 |
| user_id | uuid | ユーザーID |
| category | text | カテゴリ名 |
| grade | int | 等級 |
| question | text | 問題文 |
| answer | text | メンバーの回答 |
| score | int | 管理者が入力する点数（初期値NULL） |
| graded_at | timestamp | 採点日時 |
| submitted_at | timestamp | 提出日時（自動記録） |

### 点数の計算方法
```
合計点 = test_results.score（4択・○×）+ text_answers.score（記述式）
```
記述式が採点待ちの場合は「〇点（記述式採点待ち）」と表示。

### item_id の命名ルール
`grade{等級}_{カテゴリ}_{項目}` の形式
例：`grade1_training_phone`、`grade1_azism_team`

## 12. AIへの進め方の指示

1. **フェーズを守る** — 現在のフェーズ外の実装は提案しない
2. **コードには必ず説明をつける** — 日本語で解説する
3. **専門用語は都度説明する** — 具体例を使って説明する
4. **小さく確認しながら進める** — 小さな単位で動作確認してから次へ
5. **ユーザーに選択肢を示す** — 複数の方法がある場合はメリット・デメリットを説明する
6. **わからない点は聞き返す** — 要件が曖昧な場合は実装前に確認する
